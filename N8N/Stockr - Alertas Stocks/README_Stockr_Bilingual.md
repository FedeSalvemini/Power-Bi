# 🇪🇸 🤖 Monitor de Alertas Supply Chain (Stockr) (n8n Workflow)

**Objetivo de Negocio:** Recepción, evaluación y distribución inteligente de alertas logísticas críticas (quiebres de stock, órdenes urgentes, errores de calidad de datos) generadas por un modelo predictivo externo.

### ⚙️ Arquitectura del Flujo
![Arquitectura n8n](arquitectura.png)

### 🛠️ Nodos y Tecnologías Utilizadas
* **Webhook (POST):** Endpoint pasivo que escucha los resultados enviados desde un script externo (FastAPI/Python).
* **If (Lógica Condicional):** Nodos de ruteo que evalúan el status general (`ok` vs `error`), presencia de advertencias ETL, y la criticidad de Órdenes de Compra (OC).
* **SplitOut:** Desagregación de arrays de OCs urgentes para procesamiento individual.
* **Telegram:** Alertas segmentadas al canal de operaciones. Reporta desde resúmenes ejecutivos de KPIs (Lost Sales, Nivel de Servicio) hasta alertas críticas de reabastecimiento por SKU.

### 💡 Lógica de Implementación (JSON)
El código fuente del flujo se encuentra en el archivo `workflow.json` adjunto en este repositorio. Para auditarlo, puede importarse directamente en cualquier instancia local o cloud de n8n.

#### Resumen Técnico:
Microservicio de monitoreo para un motor de inventario. Escucha los payloads JSON emitidos por un modelo predictivo, parsea las métricas operativas clave (KPIs de supply chain), evalúa reglas de negocio (órdenes urgentes, errores de ingesta) y distribuye alertas accionables al equipo de operaciones.

# 🇬🇧 🤖 Supply Chain Alert Monitor (Stockr) (n8n Workflow)

**Business Objective:** Reception, evaluation, and intelligent distribution of critical logistics alerts (stockouts, urgent orders, data quality errors) generated by an external predictive model.

### ⚙️ Flow Architecture
![n8n Architecture](arquitectura.png)

### 🛠️ Nodes & Technologies Used
* **Webhook (POST):** Passive endpoint listening for results sent from an external script (FastAPI/Python).
* **If (Conditional Logic):** Routing nodes that evaluate the general status (`ok` vs `error`), presence of ETL warnings, and the criticality of Purchase Orders (PO).
* **SplitOut:** Disaggregation of arrays of urgent POs for individual processing.
* **Telegram:** Segmented alerts to the operations channel. Reports ranging from executive summaries of KPIs (Lost Sales, Service Level) to critical restocking alerts per SKU.

### 💡 Implementation Logic (JSON)
The source code of the flow is located in the attached `workflow.json` file in this repository. To audit it, it can be imported directly into any local or cloud n8n instance.

#### Technical Summary:
Monitoring microservice for an inventory engine. Listens to JSON payloads emitted by a predictive model, parses key operational metrics (supply chain KPIs), evaluates business rules (urgent orders, ingestion errors), and distributes actionable alerts to the operations team.
